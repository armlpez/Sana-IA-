P0 — Batch de código enforcement (los dos primeros, bajo riesgo)

Estos dos son gemelos: ambos en chat.service.ts, ambos son "hacé que el código garantice lo que el modelo debería haber hecho", ambos chicos. Van juntos en un PR.

Mejora 1 — Estado terminal inconsistente (esta subió de severidad al verificarla).
Lo que reporté antes como "solo loguea un warning" es peor de lo que pensaba. Rastreé la cadena completa:
- buildConsultationUpdate:504-505 escribe status=COMPLETED apenas el modelo dice "completed", sin chequear que haya diagnóstico.
- Pero el guardado del diagnóstico (chat.service.ts:197) está protegido por if (parsed.diagnosis).
- Resultado: consulta marcada COMPLETED, cero filas en la tabla diagnosis.
- Y reports.service.ts:88-94 exige AMBAS cosas → tira "No completed diagnosis available". El paciente ve su consulta "terminada" pero nunca puede generar el informe.

El fix es un invariante: solo promover a COMPLETED si hay diagnóstico o es una emergencia (la emergencia cierra sin RCA a propósito — emergencyThisTurn ya es parámetro de esa función). Si el modelo dice "completed" sin diagnóstico y sin emergencia, se queda en ANALYZING. ~5 líneas.

Mejora 2 — Fallback amnésico.
Confirmado en safe-fallback.builder.ts:89-93: forChat() devuelve extractedData: { symptoms: null, treatment: null, duration: null } fijo, y solo recibe emergencyDetected y kind — nunca ve los datos del paciente. Pero fijate que a 4 líneas de distancia, el mismo handleChatFailure sí recupera bien el summary: summary: consultation.summary ?? null (:310). El fix es replicar ese patrón que ya existe:
extractedData: {
  symptoms: consultation.extractedSymptoms ?? null
  treatment: consultation.extractedTreatment ?? null,
  duration: consultation.extractedDuration ?? null,
}
  treatment: consultation.extractedTreatment ?? nu
  duration: consultation.extractedDuration ?? null,
}
No es pérdida real de datos (el próximo turno relee la DB), pero el paciente ve que "se olvidó de todo" ante un error transitorio.
Fix trivial, consistente con el código existente.

P1 — Acotar el peor caso (Mejora 3)

La repetición en sí es desobediencia del modelo — el tracking determinístico funciona perfecto (askedTopics acumula bien, verificado en :522-527), pero el modelo ignora la instrucción de no repreguntar. Eso es difícil de garantizar 100% sin reescribir cómo se genera la pregunta. Lo que SÍ es código-fixable es la consecuencia: hoy no hay tope de turnos en código, por eso la prueba corrió 6 turnos cuando el diseño dice 4. Un contador determinístico (derivable del COUNT de mensajes del usuario) que fuerce el reporte final con lo que se sabe, después de N turnos, acota el peor caso aunque el modelo siga repitiendo.

P2 — Backstop de confidencialidad (Mejora 4)

Honestidad sobre la severidad: lo que se filtró (n regla de "4 turnos") es información de bajasensibilidad — no son credenciales ni PHI. No es una catástrofe. PERO el principio importa: es el único "no negociable"-adyacente sin respaldo de código. El fix es un filtro de salida que escanee el message saliente por tokens internos conocidos (FASE 0/1/2, nombres de campos) y lo reemplace por una redirección segura — exactamente el patrón de la emergencia. Defensa en profundidad.

P3 — Decisiones de producto (Mejoras 5 y 6)

La 5 no es un bug: la regla define emergencia como riesgo agudo, y el cuadro de "síntomas B" es crónico-grave. Es correcto que no dispare emergencia, pero abre la pregunta: ¿hace falta un tercer nivel "derivación urgente" entre normal y emergencia? Es diseño clínico, no ingeniería — lo decide el equipo médico (Eduardo Daka aprobó el texto de emergencia, sería su llamada). La 6 es calidad de dato menor, la dejaría para el final.
